LOCAL MULTI-MODEL AI ORCHESTRATION PLATFORM

(Offline, Modular, Specialist-Based Intelligence)

🔒 CORE DIRECTIVE (READ FIRST)

You are building a fully local, offline AI orchestration system that behaves as one unified intelligence while internally using multiple specialized AI models.

🚫 You must NOT attempt to merge model weights.
🚫 You must NOT rely on cloud APIs.
🚫 You must NOT create a monolithic LLM.

✅ You must implement model orchestration, central planning, shared memory, unified persona, and output synthesis.

This system must feel like one AI to the user.

🧩 HIGH-LEVEL ARCHITECTURE

The system consists of six mandatory subsystems:

Central Planner (AI Brain)

Task Router (Deterministic Code)

Specialist Models (Experts)

Shared Conversation Memory

Unified Persona System

Output Synthesis Layer

All subsystems must be modular, replaceable, and local-only.

🧠 1. CENTRAL PLANNER (THE BRAIN)
Purpose

The Central Planner does not solve tasks directly unless trivial.
It decides:

What the user wants

Whether reasoning is required

Which specialist models to call

The execution order

How results should be merged

Model

Use Meta LLaMA 3.1 8B (quantized, GGUF)

Rules

Planner outputs structured JSON only

Planner never produces final user-facing answers

Planner must remain stable and predictable

Mandatory Planner Prompt (System-Level)
You are a central AI planner.
You do not solve tasks directly unless trivial.
Your role is to:
- interpret user intent
- decompose tasks into steps
- select appropriate specialist models
- control execution order
- preserve a unified personality and memory

You must output structured JSON decisions only.
🔀 2. TASK ROUTER (CODE, NOT AI)
Purpose

The Task Router is pure code logic that executes planner decisions.

Responsibilities

Parse planner JSON

Call specialist models

Handle parallel or sequential execution

Manage failures and fallbacks

Return results to the synthesis layer

Rules

No AI logic here

No interpretation

Deterministic behavior only

🎭 5. UNIFIED PERSONA SYSTEM

All models must behave as one personality.

Persona Definition (Single Source of Truth)
tone: calm, technical, precise
verbosity: adaptive
values:
  - correctness
  - clarity
  - honesty
rules:
  - never contradict past answers without explanation
  - ask clarifying questions only when necessary
  - explain reasoning when beneficial
Enforcement

Inject persona into every model prompt

Output synthesis must enforce persona consistency

🧪 6. OUTPUT SYNTHESIS (FINAL VOICE)
Purpose

Merge all specialist outputs into one coherent response.

Process

Collect model outputs

Normalize formats

Resolve conflicts

Enforce persona

Produce final answer

Model Used

Central Planner model (LLaMA 3.1 8B)

⚠️ This is the ONLY component that talks to the user.
